## LibSkills Project — New Big Idea

草猛半夜（01:33-01:41）冒出一个新项目想法：**LibSkills**。

一句话：给所有开源库的AI技能做一个中心化仓库 + CLI，让AI写代码时直接读预编译的使用规范，不用每次都读源码或靠猜。

### 核心问题
- AI每次遇到不熟悉的库要么猜（产生幻觉）、要么读源码（费token）
- LibSkills把每个库的最佳实践、陷阱、红线等编码为一个skill文件，AI直接读取

### 架构设计
- CLI用Rust开发，定位为"AI的包管理器"
- 分组: `main`（事实标准库）/ `contrib`（小众库）
- 分层: `Tier 1`（官方审核）/ `Tier 2`（社区提交）
- 命名: `[language]/[author]/[name]`，无冲突时可简写
- `index.json` 和CLI放一起，registry另开目录
- 组织（Organization）vs 个人账号 → 决定**建组织**
- LICENSE → **Apache 2.0**
- 语言 → **全英文，不得出现中文**
- 草猛说"明天就开始做"

## GPT第二轮深度建议（02:44）— 协议设计层面

GPT在草猛转发了蓝图之后，又追加了28条更深的范式级思考。

### 第14条：GitHub-first vs Registry-first
- GitHub-first: skills存在git repo，CLI从GitHub拉取。简单MVP快，但不适合百万skill
- Registry-first: registry API + index cache + snapshot。可扩展，前期复杂
- GPT建议：**第一阶段GitHub-first，第二阶段再过渡到Registry-first**

### 第15条：Skill ≠ 库文档
- 不写spdlog有async logger/sink这些文档内容
- 重点写：哪些地方最容易踩坑（async生命周期crash、flush与shutdown顺序错误）

### 第16条：Skill必须能被AI"部分读取"
- 不要一个1万行markdown
- 多个小块：pitfalls.md / threading.md / ownership.md / shutdown.md
- AI可以只读threading，不用读全部

### 第17条：危险等级
- `risk_level: high | medium | low`
- high: 并发/生命周期/allocator/async runtime/unsafe
- low: utility/formatting/string helper
- AI可以优先读高风险库

### 第18条：Skill爆炸时的索引结构
- 100万skills时目录结构不重要，索引结构才是重点
- 需要global index：`{ "cpp/spdlog": { "tier": "tier1", "risk": "high" } }`

### 第19条：Skill冲突
- tier2可能出现多观点冲突（community-a说async safe，community-b说unsafe）
- 建议：**允许冲突存在**，不要试图统一世界

### 第20条：Skill可组合
- tokio + reqwest 组合使用
- `{ "requires": ["tokio"] }` 让AI自动补齐上下文

### 第21条：Skill完成标准
- 最低要求：summary + lifecycle + pitfalls + examples + version + constraints

### 第22条：Skill Review System
- 类似Rust RFC / Arch Wiki / Linux Kernel Docs
- 不能人人都能改tier1

### 第23条：最重要的数据不是skill文件，是使用数据
- 哪些skill最常被调用
- 哪些skill最容易出错
- 形成feedback loop

### 第24条：真正的壁垒是Skill accumulation
- 别人可以复制CLI
- 但复制不了数十万高质量经验

### 第25条：Slogan
- 不好：AI library helper

- 更强：Make AI understand libraries safely

### 第26条：Skill Graph（知识图谱）
- spdlog → fmt → allocator → thread pool
- AI自动理解依赖关系

### 第27条：Skill Pack
- `backend-pack` → 自动装postgres/redis/grpc/logging/metrics全套skills

### 第28条：现在最重要的不是写代码，是定义世界观

- 必须清楚LibSkills不是什么

## GPT第三轮宇宙规则（02:49）— 定义LibSkills的最小单位与世界观

### 第29条：最小单位不是"一个库=一个skill"
- 建议：**一个库 = 一个Skill Namespace**
- 内部按功能拆：logging/async/sink/formatter/rotation
- AI不需要整个库，更需要局部知识

### 第30条：Skill必须可组合
- AI想做"异步日志+文件轮转+多线程"
- 组合读取：spdlog/async + spdlog/rotation + spdlog/threading

### 第31条：三层结构
- Library（库级）→ Capability（功能级）→ Constraint（行为级）
- 例：spdlog → async logging → must flush before destroy

### 第32条：AI何时触发Skill
- 自动触发条件：detect `import spdlog` / `cargo add tokio` / `pip install requests`
- 自动preload skill

### 第33条：知识生命周期（时间维度）
- 不只是文件生命周期
- 需要版本时间线：`introduced_in`, `deprecated_in`

### 第34条：如何避免污染
- tier2有观点冲突/错误经验/偏见
- 需要confidence分数

### 第35条：冲突允许存在
- 社区认为tokio safe，官方认为unsafe in nested runtime
- 多来源并存，不要统一

### 第36条：Skill的三种消费模式
- quick safety：只读constraints + pitfalls
- integration：读examples + lifecycle + dependencies
- deep understanding：读全部

### 第37条：Skill Compression（压缩）
- 未来skill很多时，需要distilled skill
- 例如"Top 10 mistakes"而不是50页知识

### 第38条：Skill继承
- redis-py 继承 redis 的知识
- 避免重复

### 第39条：实验性Skill
- stability: experimental 标记
- 否则未来难扩展

### 第40条：核心不是知识存储，是知识提炼
- 文档=原始矿石，Skill=精炼钢铁

### 第41条：Skill的原子单位是knowledge atom
- 不是markdown
- `{ type: "constraint", content: "must shutdown before destroy" }`
- 这是未来真正的核心

### 第42条：定义LibSkills不是什么
- NOT: 文档镜像/API docs/教程站/wiki/StackOverflow
- IS: 行为知识层（Behavioral Knowledge Layer）

## GPT第五轮 — 分布式Skill来源（02:54）

### 核心观点：不要锁死在LibSkills仓库
三种来源：
1. Registry Skill — 官方中心仓库（LibSkills/libskills-registry）
2. Repository Skill — 库作者自带（.libskills/ 或 .github/libskills/）
3. Local Skill — 企业/个人私有（~/.libskills/private/）

### LibSkills的定位应该是：Registry + Spec + Resolver
- Registry: 中心仓库
- Spec: 格式定义
- Resolver: 自动发现Skill（libskills resolve spdlog → 本地→registry→repo→mirror）

### 关键原则
- 不要害怕别人自己维护——真正的价值不在于谁拥有文件，而在于谁定义标准
- 类比OpenAPI（接口文件到处是，但OpenAPI定义规则）
- "任何仓库都能携带Skill" → LibSkills从项目变成标准

### 未来功能
- libskills discover — 自动扫描package.json/Cargo.toml/CMakeLists.txt并发现依赖对应Skill

## GPT第六轮 — 协调层与知识演化（02:56）

### 定位转变：从内容拥有者 → 可信协调层
- 未来spdlog自己维护官方skill、社区20个优化版、企业内部安全版
- LibSkills的角色不是"唯一正确仓库"，而是让skill可被发现/比较/排序/信任
- 核心资产变成：Resolver + Ranking + Trust
- 类比DNS：你不需要拥有内容，你需要解析到正确的内容

### Skill漂移问题
- 2026年的best practice到2028年可能完全变了
- Skill不是静态文件，是持续演化的认知
- 需要支持skill timeline：spdlog 1.8 / 1.10 / 1.14 各自的最佳实践
- AI最容易错的地方：用旧经验解释新版本

### Skill Authoring Language
- 人类写文档喜欢解释/教学/展示
- Skill需要的是约束/风险/条件/失败模式 —— 这是完全不同的写作方式
- 可能需要一种"逼迫作者表达限制条件"的格式

### 最终定位
- 不是"让AI更聪明"
- 而是"让AI不犯错"
- 聪明容易被模型升级替代，不犯错是长期问题

## GPT第七轮 — 认知压缩与信息架构（02:57）

### LibSkills的本质是"认知压缩"
- 一个人用spdlog需要经历：README → Wiki → Issues → StackOverflow → 博客 → benchmark → crash → 实战经验
- 才得出"async logger生命周期不能乱来"
- LibSkills把几十小时经验压缩成一个可读取对象
- 软件工程最大的成本不是写代码，是理解成本

### LibSkills的真正单位可能不是"库"，而是"模式"
- 库层：spdlog / tokio / redis / grpc
- 模式层：async lifecycle / ownership transfer / resource pool / backpressure / shutdown ordering
- 库会变化，模式不会
- async runtime生命周期在Rust/C++/Python/Node.js中都存在
- 最终可能是 Software Behavior Knowledge，不只是 Library Skills

### LibSkills是过滤器，不是仓库
- AI的问题不是没知识，是知识太多
- LibSkills是高信号过滤层：不要读全部，只读最危险/最昂贵/最容易错/最难发现
- 未来值钱的不更多数据，是更少但更正确的数据
- Skill本质不是内容，是选择——哪些知识重要、优先、可信、应该忽略

### 总结定位演变
- 第一阶段：技能仓库
- 第二阶段：知识包管理器
- 第三阶段：认知压缩层
- 第四阶段：软件行为知识
- 本质：信息架构问题，不是知识仓库问题

## GPT第八轮 — 项目哲学深层（03:04）

### 核心哲学：降低错误的代价
- 代码越来越容易生成，但理解越来越昂贵
- 以前难的是"写"，现在难的是：判断/集成/约束/生命周期/依赖行为/隐式规则
- LibSkills不是"让AI更会写代码"，是"让AI更少误解世界"
- 幻觉本质不是乱编，是缺少可信上下文

### 过滤哲学，不是堆积哲学
- 世界不缺信息，缺"知道什么重要"
- 对于spdlog：logger->info()不重要，生命周期顺序/async销毁风险/flush时机重要
- 不是知识数量决定质量，是知识排序决定质量
- LibSkills是重要性引擎，不是知识库

### 文档 vs Skill
- 文档是"描述"：这个库能做什么
- Skill是"提醒"：这个库哪里容易死
- 源码→文档→Skill = 能做什么→怎么做→哪里不能错（清晰三层）

### 只记录"高代价知识"
- 不知道会导致：crash/泄漏/错误架构/并发问题/生命周期问题/性能灾难/难以排查bug
- Skill必须是经验压缩，不是知识复制
- 知识本来就存在，真正稀缺的是被验证过的重要经验

### 不追求完整，追求"足够正确"
- 完整=冗余
- AI不需要知道所有，只需要避免关键错误

### 预理解 / 风险感知层
- 传统AI流程：遇到库→临时学习→猜测→生成
- LibSkills：遇到库→已有经验模型→生成
- 专家不是知道更多，是知道哪些地方危险
- LibSkills不是知识仓库，是风险感知层

## 最终总结（03:14）- LibSkills项目完整定位

### 一句话
LibSkills = 开源库行为知识层（Behavioral Knowledge Layer for Open-Source Libraries）

### 最终形态
